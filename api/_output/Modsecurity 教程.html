5、编写简单的黑名单规则

在后续教程中，我们将嵌入 OWASP ModSecurity 核心规则，这是一个全面的规则集合。但对我们来说，首先学习如何自己编写规则很重要。

让我们举一个简单的例子：服务器阻止特定 URI 的访问。我们用 HTTP 403 状态码来响应此类请求。我们为 ModSecurity 编写规则，并为其分配 ID 10000。

SecRule  REQUEST_FILENAME "/phpmyadmin" "id:10000,phase:1,deny,log,t:lowercase,t:normalisePath,\
  msg:'Blocking access to %{MATCHED_VAR}.',tag:'Blacklist Rules'"

7、编写简单的白名单规则

使用步骤 7 中描述的规则，我们能够阻止对特定 URL 的访问。现在使用相反的想法：希望只能访问特定的 URL。此外，我们将只接受已知的特定格式的 POST 参数。

不再是我们试图在用户提交的内容中查找已知攻击，而是必须由用户证明他的请求符合我们的所有标准。

我们的示例是登录的白名单，其中有表单，提交凭证，注销等。

SecMarker BEGIN_WHITELIST_login

# 拒绝在 URI 中连续包含两个点的请求
# Make sure there are no URI evasion attempts
SecRule REQUEST_URI "!@streq %{REQUEST_URI_RAW}" \
    "id:11000,phase:1,deny,t:normalizePathWin,log,\
    msg:'URI evasion attempt'"

# 检测路径
# START whitelisting block for URI /login
SecRule REQUEST_URI "!@beginsWith /login" \
    "id:11001,phase:1,pass,t:lowercase,nolog,skipAfter:END_WHITELIST_login"
SecRule REQUEST_URI "!@beginsWith /login" \
    "id:11002,phase:2,pass,t:lowercase,nolog,skipAfter:END_WHITELIST_login"

# Validate HTTP method
SecRule REQUEST_METHOD "!@pm GET HEAD POST OPTIONS" \
    "id:11100,phase:1,deny,log,tag:'Login Whitelist',\
    msg:'Method %{MATCHED_VAR} not allowed'"

# Validate URIs
SecRule REQUEST_FILENAME "@beginsWith /login/static/css" \
    "id:11200,phase:1,pass,nolog,tag:'Login Whitelist',\
    skipAfter:END_WHITELIST_URIBLOCK_login"
SecRule REQUEST_FILENAME "@beginsWith /login/static/img" \
    "id:11201,phase:1,pass,nolog,tag:'Login Whitelist',\
    skipAfter:END_WHITELIST_URIBLOCK_login"
SecRule REQUEST_FILENAME "@beginsWith /login/static/js" \
    "id:11202,phase:1,pass,nolog,tag:'Login Whitelist',\
    skipAfter:END_WHITELIST_URIBLOCK_login"
SecRule REQUEST_FILENAME \
    "@rx ^/login/(displayLogin|login|logout).do$" \
    "id:11250,phase:1,pass,nolog,tag:'Login Whitelist',\
    skipAfter:END_WHITELIST_URIBLOCK_login"

# If we land here, we are facing an unknown URI,
# which is why we will respond using the 404 status code
SecAction "id:11299,phase:1,deny,status:404,log,tag:'Login Whitelist',\
    msg:'Unknown URI %{REQUEST_URI}'"

SecMarker END_WHITELIST_URIBLOCK_login

# Validate parameter names
SecRule ARGS_NAMES "!@rx ^(username|password|sectoken)$" \
    "id:11300,phase:2,deny,log,tag:'Login Whitelist',\
    msg:'Unknown parameter: %{MATCHED_VAR_NAME}'"

# Validate each parameter's cardinality
SecRule &ARGS:username  "@gt 1" \
    "id:11400,phase:2,deny,log,tag:'Login Whitelist',\
    msg:'%{MATCHED_VAR_NAME} occurring more than once'"
SecRule &ARGS:password  "@gt 1" \
    "id:11401,phase:2,deny,log,tag:'Login Whitelist',\
    msg:'%{MATCHED_VAR_NAME} occurring more than once'"
SecRule &ARGS:sectoken  "@gt 1" \
    "id:11402,phase:2,deny,log,tag:'Login Whitelist',\
    msg:'%{MATCHED_VAR_NAME} occurring more than once'"

# Check individual parameters
SecRule ARGS:username "!@rx ^[a-zA-Z0-9.@-]{1,32}$" \
    "id:11500,phase:2,deny,log,tag:'Login Whitelist',\
    msg:'Invalid parameter format: %{MATCHED_VAR_NAME} (%{MATCHED_VAR})'"
SecRule ARGS:sectoken "!@rx ^[a-zA-Z0-9]{32}$" \
    "id:11501,phase:2,deny,log,tag:'Login Whitelist',\
    msg:'Invalid parameter format: %{MATCHED_VAR_NAME} (%{MATCHED_VAR})'"
SecRule ARGS:password "@gt 64" \
    "id:11502,phase:2,deny,log,t:length,tag:'Login Whitelist',\
    msg:'Invalid parameter format: %{MATCHED_VAR_NAME} too long (%{MATCHED_VAR} bytes)'"
SecRule ARGS:password "@validateByteRange 33-244" \
    "id:11503,phase:2,deny,log,tag:'Login Whitelist',\
    msg:'Invalid parameter format: %{MATCHED_VAR_NAME} (%{MATCHED_VAR})'"

SecMarker END_WHITELIST_login

由于这是一组多行规则，我们使用两个标记分隔规则组：BEGIN_WHITELIST_login 和 END_WHITELIST_login。

SecMarker 的作用是什么？

为规则创建一个固定的标记，然后可根据标记使用 skipAfter 跳转至指定的规则。 SecMarker 指令基本上创建了一个不做任何事情的规则，其唯一目的是携带已给定的 ID。

第一条规则 (ID 11000) 强制执行我们的策略，以拒绝在 URI 中连续包含两个点的请求。连续的两个点可能是逃避我们后续路径标准的一种方式。例如构建一个看起来像访问其他文件夹的 URI，但随后使用 .. 逃离该文件夹并访问 /login 。此规则确保这些游戏都不能在我们的服务器上玩。

在以下两个规则（ID 11001 和 11002）中，我们检查我们的规则集是否受到影响。如果路径以小写和规范化方式不以 /login 开头，我们会跳到结束标记 - 在日志文件中没有条目。

专注于登录的想法，即登录页面将由匿名用户访问。一旦他们执行了登录，至少已经证明了他们的凭据并建立了一定的信任。因此，登录可能是匿名攻击者的目标，我们希望很好地保护它。

首先，我们查看规则 ID 11100 中的 HTTP 方法。显示登录表单将是一个 GET 请求；提交凭据将是一个 POST 请求。一些客户端也喜欢发出 HEAD 和 OPTIONS 请求，允许这些请求不会造成太大伤害。其他内容，PUT 和 DELETE 以及所有 webdav 方法都将被规则阻止。我们使用并行匹配运算符 (@pm) 检查四个列入白名单的方法。这比正则表达式要快，而且可读性也更高。

在从规则 ID 11200 开始的规则块中，我们详细检查了 URL。我们建立三个文件夹，允许访问静态文件：/login/static/css/、/login/static/img/ 和 /login/static/js/。我们不想对文件夹中下所有单个文件进行细粒度的管理，因此我们只允许访问这些文件夹。

规则 ID 11250 不同。它定义了用户动态请求的目标。我们构造了一个正则表达式，它允许三个 URI：/login/displayLogin.do、/login/login.do 和 /login/logout.do。此列表之外的任何内容都将被禁止。

但是这一切是如何检查的呢？规则 11200、11201、11202 和 11250 检查 URI。

如果有匹配项，我们不会阻止，而是跳转到标签 END_WHITELIST_URIBLOCK_login。

当我们到达这个标签时，我们知道 URI 是预定义集合之一：请求符合我们的规则。

但是如果我们传递了 11250 并且仍然没有命中 URI，那么我们知道客户端看起来像一个罪犯，我们可以相应地阻止它。这是在 ID 为 11299 的回退规则中执行的。请注意，这不是一个条件 SecRule，而是一个 SecAction，它是同一件事，但没有运算符和参数。这些操作会立即执行，目的是阻止请求。这里有一个转折：如果我们在这条规则中阻止，我们不会告诉客户端他的请求被禁止（HTTP 状态 403，这将是拒绝的默认值）。相反，我们返回一个 HTTP 状态 404，让客户端对我们的规则集的存在一无所知。

默认情况下，ModSecurity 核心规则集使用评分机制。对于请求违反的每条规则，都会增加一个分数。当所有请求规则都通过时，将分数与限制进行比较。如果达到限制，则请求被阻止。同样的事情发生在响应中，我们希望避免信息泄露给客户端。

默认情况下，核心规则集处于阻止模式。如果违反规则并且分数达到限制，则阻止将立即生效。

第二条规则 id 900000 将 Paranoia Level 定义为 1。

核心规则在 paranoia 级别 1 - 4 分为四组。顾名思义，paranoia 级别越高，规则越偏执。默认值为偏执级别 1，其中规则非常理智，并且很少出现误报。

当您将 paranoia level 提高到 2 时，将启用其他规则。从 paranoia level 开始，您将面临越来越多的误报

这个数字会随着 PL3 的提高而提高。我们将在后续的章节中处理误报。

您可以使用偏执级别设置来控制规则集的攻击性，并且 PL3 和 PL4 确实适用于具有非常高安全性的高级用户需要。

4、2 规则编写

规则概览如下：

SecRule REQUEST_FILENAME "/phpmyadmin" "id:10000,phase:1,deny,log,t:lowercase,t:normalisePath,\

 msg:'Blocking access to %{MATCHED_VAR}.',tag:'Blacklist Rules'"

上述规则可拆分为几部分：

1） SecRule

SecRule：开始规则

2） REQUEST_FILENAME

REQUEST_FILENAME：此变量包含不带查询字符串部分的相对请求 URL。

REQUEST_FILENAME 变量检查请求的路径。如果 /phpmyadmin 出现在此路径中的任何位置，我们希望在第一个处理阶段中立即阻断它。

TODO：常用的变量还有哪些？？？对应的代码在什么地方？

3） phase 阶段

phase: 选择规则在哪个阶段执行，或者可在 SecDefaultAction 中创建默认值

  • Request headers (REQUEST_HEADERS)

完成读取请求头（post-read-request 阶段）后立即处理此阶段中的规则。 此时尚未读取请求体，

  • Request body (REQUEST_BODY)

该阶段是通用输入分析阶段。在此阶段，您已收到请求参数，大多数面向应用的规则在这里执行。

要访问 Request Body 阶段的数据，必须将 SecRequestBodyAccess 设置为 On。

  • Response headers (RESPONSE_HEADERS)

该阶段发生在响应头被发送回客户端之前。

  • Response body (RESPONSE_BODY)

该阶段是通用输出分析阶段。此时可运行规则检查响应体。

要访问 Response Body 阶段数据，必须将 SecResponseBodyAccess 设置为 On

  • Logging (LOGGING)

该阶段在日志记录发生之前运行。进入此阶段的规则只会影响日志记录的执行方式。此阶段可用于检查服务器记录的错误消息。您不能在此阶段拒绝 / 阻断连接，因为为时已晚。

4） deny

停止规则处理，并拦截事务。

所属组：阻断性动作

5）log 字段

log: 记录日志

6） t:lowercase,t:normalisePath

我们的路径标准以小写字母维护。因为我们使用了 t:lowercase 转换，所以我们能捕获路径中所有可能的小写和大写组合。

normalisePath 同 normalizePath，

确保在应用规则之前，路径已经被转换。

7） msg 消息

msg: 在 msg 中填一条信息，如果触发规则，它将显示在服务器的错误日志中。

8） tag 标签

tag: 标签，用于将不同的规则进行分组。

4、4 检测 Json 格式数据

默认情况下，ModSecurity 将使用 URLENCODED 和 MULTIPART 解析器分别处理 application/x-www-form-urlencoded 和 multipart/form-data 类型的请求体，如要对 JSON 格式的数据进行解析并检测，需要临时更改解析器，如下方规则所示：

SecRule REQUEST_HEADERS:Content-Type "application/json" \
    "id:100,phase:1,t:none,t:lowercase,pass,nolog,ctl:requestBodyProcessor=JSON"

上述规则表示，如果请求头中 Content-Type 的值为 "application/json"，则临时将请求体的解析器改为 JSON 解析器。

此时，我们可以修改 Content-Type 为 "application/json"，然后 POST 以下数据到服务器进行测试：

{
    "Image": {
        "Width": 800,
        "Height": 600,
        "Title": "View from 15th Floor",
        "Thumbnail": {
            "Url": "http://www.example.com/image/481989943",
            "Height": 125,
            "Width": "100"
        },
        "IDs": ["116", "943", "234", "ABC"]
    }
}

能够成功解析出 json 数据后，就开始编写过滤规则了。

#对Image.Width的值进行判断，如果大于200，则阻断此次访问
SecRule ARGS:Image.Width "@gt 200" "phase:2,log,id:101,deny,msg:'Json param image.width gt 200'"

#对Image.IDs集合中的值进行判断，如果存在不是数字的参数，则阻断此次访问
SecRule ARGS:Image.IDs.IDs "!@rx ^[0-9]*$" "phase:2,log,id:102,deny,msg:'ID is not number'"

4、6 加载远程机器的规则

当需要更新规则的 waf 节点居多时，比如成千上万时，需要将新的规则文件同步更新到所有的 waf 节点上，对于运维人员是个很大的工作量。敲好 ModSecurity 支持加载远程机器的规则：

SecRemoteRules 允许从指定的某个 HTTPS 站点上的文件来加载规则，语法如下：

SecRemoteRules some-key https://www.yourserver.com/plain-text-rules.txt

some-key 意味着可以定义一个 key 值，远程服务器可以根据 key 值，来判断要返回哪些规则内容，或是否返回规则内容，当然，随便定义一个 key 值，远程服务器不进行任何判断也可以。

使用 SecRemoteRules 后，WEB 服务在启动时，ModSecurity 会访问配置的 HTTPS 路径来读取远程文件中的规则，而与此同时，请求头中会增加三个参数：ModSec-unique-id、ModSec-status、ModSec-key，如下所示：

GET /plain-text-rules.txt HTTP/1.1
User-Agent: modesecurity
Host: www.yourserver.com
Accept: */*
ModSec-unique-id: e781ee72979d16cd91ed89c716dc087f195d7cb8
ModSec-status: 2.9.3,(null),1.5.2/1.5.2,8.43/8.43 2019-02-23,Lua 5.1,2.9.9,e781ee72979d16cd91ed89c716dc087f195d7cb8
ModSec-key: MkIe2c8pIK8AI2g5mXipvl4x

其中，ModSec-key 的值就是定义的 key 值，在实际中我将其设置为了 MkIe2c8pIK8AI2g5mXipvl4x。

4、7 ModSecurity OWASP 规则集的两种配置模式

4、7、1 异常评分模式（默认）

在此模式下，当检测到威胁时，并不会直接阻断此次访问，而是向下继续进行规则匹配，每个匹配成功的规则都会增加 "异常分数"，在对客户端发起的请求数据检测结束时，以及对返回数据检测结束时，都会对异常分数的总和进行判断，如果大于设置的阈值，才会进行阻断动作，并向客户端返回 403 代码，审计日志中也会记录此次访问中所有匹配成功的规则信息。

4、7、2 独立控制模式

此模式是 V2 版本规则集的默认模式，它的工作方式是，只要有一条规则匹配成功，便拦截此次访问，审计日志中也只会记录第一次检测到威胁的规则信息。

4、7、3 优缺点

异常评分模式：由于每次请求都会匹配所有规则，因此在高并发情况下，效率相对较低，服务器资源占用较高，但误报率相对较低；

独立控制模式：检测到一次威胁就直接阻断请求，因此在高并发情况下，效率相对较高，服务器资源占用较小，但误报率相对较高。除此之外该模式还有一个优点，即可以通过全局配置，设置当访问被拦截后，跳转到自定义的提示页面。
